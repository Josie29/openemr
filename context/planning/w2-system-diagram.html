AgentForge Clinical Co-Pilot · System Topology

How OpenEMR and the agent service connect

Independently deployed services talk over HTTP. The agent reads the chart through a patient-scoped SMART token; it holds no write path and no database. Writes go back to OpenEMR from the browser under the clinician's own session — through the PHP app, into a separate managed database.

A
SMART launch
Hidden-iframe EHR launch (OAuth2 / PKCE) against OpenEMR. The sidebar receives a patient-scoped read token; the token's own patient claim is the only scope.
B
Chat & read
Sidebar → agent /chat with Authorization: Bearer. The agent reads /fhir/* with that token, calls Qdrant + Mistral + the LLM, and answers with derived_facts.
C
Write-back
The browser posts derived_facts to persist-facts.php — same-origin, session cookie + CSRF, not the agent. It writes under the clinician's own ACL.

The read path goes through the agent

Browser → Agent → OpenEMR FHIR

The agent builds a per-request, read-only FHIR client bound to the inbound SMART token. Its FHIR client has no POST or PUT — it can only read the chart. It never holds a database credential.

The write path bypasses the agent

Browser → OpenEMR PHP → MySQL

No patient- or system-scoped write surface the agent could reach exists in this fork. So the browser posts the extracted facts to a session-authed OpenEMR endpoint, which writes via the service layer into the separate MySQL service — the physician is the write principal, and the app it re-clones is disposable while the data survives.

Runtime · one /chat turn

How a turn runs — the supervisor graph

Inside copilot-agent, a supervisor routes each turn to workers, loops until it has enough, then composes a grounded answer. Every worker output passes a grounding gate. The agent only reads — the chart write happens two steps later, in the browser.

Tools, by worker — and what each reads

Tool(s)OwnerReachesR / W
get_patient · get_problems · get_medications · get_allergies · get_encounters · get_encounter_note intake-extractor OpenEMR FHIR read
list_documents intake-extractor OpenEMR FHIR (DocumentReference discovery) read
attach_and_extract intake-extractor FHIR Binary → Mistral OCR · fills deps.extractions read (no chart write)
search_guidelines evidence-retriever Qdrant hybrid RAG + Cohere rerank read

The answerer has no data tools — it may only restate claims the workers already grounded. No tool anywhere writes to the chart.

When the write happens — two steps behind the graph

The graph is read-only and self-contained. The write is a different process, after the answer is already grounded.

1 · in the graph

OCR fills the sidecar of facts

attach_and_extract runs Mistral OCR and stores the typed result in deps.extractions. Nothing is written to the chart.

read-only
2 · after the gate

Facts ride the response

Once the answer passes grounding, _answer_payload projects the extractions into derived_facts on the /chat JSON. Still no write.

agent boundary
3 · outside the agent

The browser writes it

The sidebar POSTs each group to persist-facts.php → OpenEMR write. CSRF + ACL (patients/docs + med), pid from the session.

the write

Every worker output and the final answer pass the same grounding gate (@agent.output_validator): a claim whose citation can't be resolved against what was actually read triggers a ModelRetry — re-ground, drop it, or say it's unavailable. Exhaust the retries and the turn degrades to a refusal, never an unsourced answer. That gate is why derived_facts is safe to write: it only carries what OCR actually extracted, and it leaves the agent only after the answer built from it was verified.

The grounding gate · why Pydantic AI

How grounding works — verification, not an LLM judge

The model writes a claim and a citation pointer. A gate then resolves that pointer against the data actually fetched this turn — with plain Python, no second model. A claim that doesn't resolve is sent back for one ModelRetry; run out of retries and the turn refuses rather than ship an unsourced answer.

The three situations it runs in

Same gate, bound to a different resolver on each agent — a worker can only ground against what it was allowed to read.

Situation (@agent.output_validator)ResolverGrounds against
intake-extractor CompositeResolver(fetched, documents) FHIR records it read + facts it OCR'd
evidence-retriever chunks (ChunkRegistry) guideline snippets it retrieved
final answerer CompositeResolver(fetched, chunks, documents) all three — can only restate what the workers grounded

The composite tries each source in order, first hit wins — and can't collide, because document-fact ids are docid#ordinal, never a FHIR uuid.

What's compared — and whether a model decides it

Citation kindWhat the model writesHow it's checkedDetermined by
Structured field (FHIR record) type · id · field objects.get((type,id)) then getattr(field) is non-None deterministic — dict + getattr
Free-text quote (note / guideline) type · id · quote whitespace-normalized substring: norm(quote) in norm(text) deterministic — string match
paraphrase / wrong case fails
Document fact (OCR'd) type · docid#ordinal id lookup + resource_type must match the recorded fact deterministic — dict lookup
The value shown to the physician — left empty — copied from the resolved record / chunk / fact and stamped on deterministic — stamped, never model-authored
A claim that fails to resolve — ModelRetry: re-ground, drop it, or say unavailable the model — the only model step; ≤ 2, then refusal
The model writes the pointer; the system fills the value. A claim's value, label, date — and every click-to-source field (document_id, page, bounding_box) — are stamped from the resolved source, unconditionally overwriting anything the model put there. So a model can cite a real record but cannot fabricate what it says, and cannot forge a source box. Grounding checks that the pointer resolves; it never trusts a model-authored value.
Why this shaped the framework choice. The gate is a per-agent pre-return validate-then-retry hook — Pydantic AI's @agent.output_validator raising ModelRetry. The decision doc calls it “the crown jewel … the deterministic half of the verification strategy.” LangGraph has no first-class pre-return validate-then-retry primitive (you rebuild the gate); the OpenAI Agents SDK's output guardrails halt on a tripwire and don't feed the correction back to the model. Keeping this mechanism byte-for-byte is why Pydantic AI won — routing was the tie-breaker it could already satisfy, not the deciding factor.

Write-back · by document type

Landing a fact and flagging it are two different things. All three write correctly, but the “not clinician-confirmed” marker is visible to a clinician in only two native screens — the report Status on Patient Results (labs) and the Verification Status field in Edit Issue (allergies). The Dashboard cards show none of the markers, and a medication's proposal intent never appears in the legacy UI at all. So a derived fact can sit on the summary looking like any other. That gap is the case for phase 2: an explicit accept step in the Co-Pilot's own sidebar, where “derived, unconfirmed” is unambiguous — and where accepting flips preliminary → final and unconfirmed → confirmed on the same rows.
The Co-Pilot's own provenance surface is separate. Clicking a citation opens source-view.php as its own chart tab (“Source: …”), rendering the source PDF with the extracted value boxed. That's the Co-Pilot's surface, distinct from the native Dashboard, Issues, and Procedures screens above — none of which link to it. It's how a physician walks a written fact back to the page it came from.
